## 1. 整体架构

```
手机 App
    │ HTTP + WebSocket
    ▼
Nginx（反向代理）
    │ 可负载均衡到多台实例
    ▼
Spring Boot 实例（可多台）
    ├── HikariCP（DB 连接池，max 50）
    ├── Lettuce（Redis 连接池，max 20）
    ├── @Async 线程池（core 20 / max 100）
    └── @Scheduled 线程池（5 个独立线程）
    │
    ├──────────────────────────┐
    ▼                          ▼
  MySQL（业务持久化）        Redis（单机）
                               ├── 位置 / GEO
                               ├── 派单队列 / 状态
                               ├── Profile 缓存
                               ├── 分布式锁
                               ├── 通知模板缓存
                               ├── 限流计数器
                               ├── WebSocket Pub/Sub
                               └── 安全（JWT黑名单、验证码、登录锁）
```

## 2. Redis Key 全景

### 2.1 位置与地理索引

| Key | 类型 | TTL | 写入时机 | 用途 |
|-----|------|-----|----------|------|
| `vol:loc:{userId}` | String (JSON) | 30s | 志愿者 WebSocket 上报 / REST | 志愿者实时位置 + 在线状态，30s 未更新视为离线 |
| `blind:loc:{userId}` | String (JSON) | 30s | 盲人 WebSocket 上报 / REST | 盲人实时位置 |
| `volunteers:geo` | GEO | 无（懒清理） | 志愿者上报位置时同步写入 | 地理空间索引，支持 GEORADIUS 近邻查询；启动时清空，志愿者重连后自动重建 |

### 2.2 串行派单流程

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `order:dispatch:queue:{orderId}` | List | 无 | 当前轮候选志愿者 ID 队列，LPOP 取下一个 |
| `order:dispatch:current:{orderId}` | String | 30s | 当前等待响应的志愿者 ID；**TTL 到期即视为超时**，DispatchScheduler 检测后自动推进 |
| `order:dispatch:round:{orderId}` | String | 无 | 当前派单轮次（1/2/3 对应 5/10/20km） |
| `order:dispatch:lock:{orderId}` | String | 短暂 | 派单操作互斥锁，防止并发写队列 |

### 2.3 志愿者 Profile 缓存

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `vol:profile:{userId}` | String (JSON) | 10min | 派单评分所需字段（评分、接单率、配速偏好等）；更新/审核时主动失效 |

### 2.4 分布式调度锁

| Key | 类型 | TTL | 对应任务 |
|-----|------|-----|---------|
| `scheduler:lock:dispatchQueue` | String | 10s | DispatchScheduler（每 5s）|
| `scheduler:lock:emergencyTimeout` | String | 15s | TimeoutScheduler 紧急事件（每 10s）|
| `scheduler:lock:rematchTimeout` | String | 15s | TimeoutScheduler 重新匹配（每 10s）|
| `scheduler:lock:matchTimeout` | String | 15s | TimeoutScheduler 匹配超时（每 10s）|
| `scheduler:lock:overdueOrders` | String | 120s | TimeoutScheduler 超时挂起（每 60s）|
| `scheduler:lock:autoComplete` | String | 120s | OrderTimeoutScheduler 自动完成（每 60s）|

### 2.5 Spring Cache（通知模板）

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `notificationTemplates::{eventType}_{targetRole}` | Hash | 30min | 通知模板；管理员更新时 @CacheEvict 主动失效，TTL 兜底保证最终一致 |

### 2.6 安全与访问控制

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `jwt:blacklist:{userId}` | String | Token 剩余时间 | 登出/封号后 Token 黑名单 |
| `sms:code:{phone}` | String (JSON) | 5min | 短信验证码（含尝试次数，超 5 次拒绝） |
| `cs:login:attempts:{username}` | String | 15min | CS 管理员连续登录失败计数 |
| `cs:login:lock:{username}` | String | 15min | CS 账号锁定（5 次失败后触发） |
| `rate_limit:{type}:{ip}` | String | 窗口秒数 | API 限流计数（auth:10/min，registration:20/min，general:60/min） |

### 2.7 业务去重

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `proximity:notified:{orderId}` | String | 24h | 邻近感知通知去重，防止重复推送 |
| `emergency:cooldown:{userId}` | String | 5min | 紧急求助冷却，防止误触连续触发 |

### 2.8 WebSocket 跨实例

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `ws:messages` | Pub/Sub Channel | — | 跨实例 WebSocket 消息广播；USER 类型路由到目标用户，CS_BROADCAST 广播给所有 CS 实例 |

## 3. 高并发优化详解

### 3.1 派单近邻查询：O(N) SCAN → O(nearby) GEORADIUS

**优化前**：`SCAN 0 MATCH vol:loc:* COUNT 1000`，扫描 Redis 所有键，复杂度 O(全部志愿者数)。志愿者越多越慢，高并发时大量 SCAN 会造成 Redis 阻塞。

**优化后**：`GEORADIUS volunteers:geo {lng} {lat} {radius} km`，只返回指定半径内的志愿者，复杂度 O(N+log N)（N 为结果数，而非总数）。

```
上报位置 → GEOADD volunteers:geo
派单时   → GEORADIUS → 只拿附近 ID → MGET vol:loc:* 拉取详情
断线时   → ZREM volunteers:geo（从 GEO 集合移除）
启动时   → DEL volunteers:geo（清空，防止重启脏数据）
懒清理   → 每次 GEORADIUS 后检查 vol:loc TTL，过期的从 GEO 集合移除
```

降级兜底：GEO 集合不存在时自动回退到全量 SCAN，保证单机测试环境可用。

---

### 3.2 Profile 批量缓存：N 次 DB 查询 → 1 次 MGET

**问题**：派单评分需要访问候选志愿者的评分、接单率、配速偏好等字段。如果每个志愿者单独查 DB，100 个候选人 = 100 次查询。

**优化**：

```
1. GEORADIUS 返回 [id1, id2, ..., idN]
2. MGET vol:profile:id1 vol:profile:id2 ... vol:profile:idN  ← 1次 Redis 请求
3. 命中的直接用，未命中的（cache miss）批量查 DB：
   SELECT * FROM volunteer_profile WHERE user_id IN (miss_ids)
4. 查到后写回 Redis（TTL 10min）
```

**缓存失效点**（任何一处触发都会立即清除对应 profile 缓存）：

| 触发场景 | 失效位置 |
|--------|---------|
| 志愿者更新个人资料 | `VolunteerService.updateProfile()` |
| 志愿者上传资质证书 | `VolunteerService.submitVerification()` |
| 管理员审核身份证 | `AdminVolunteerService.reviewIdCard()` |
| 管理员审核资质证书 | `AdminVolunteerService.reviewCertificate()` |
| 志愿者接单 | `DispatchService.handleAccept()` |
| 志愿者拒单 | `DispatchService.handleDecline()` |
| 志愿者响应超时 | `DispatchService.handleVolunteerTimeout()` |
| 盲人提交评价 | `ReviewService.createReview()` |

### 3.3 原子统计更新：读-改-写 → 单条 SQL

**问题**：接单/拒单/评分时需要更新 `totalAccepted`、`acceptanceRate`、`avgRating` 等字段。原来的读-改-写三步操作在多实例并发时会发生数据竞争（两个实例同时读到相同旧值，各自 +1，实际只增加了 1）。

**优化**：全部改为 native query，数据库层原子执行 + `ROUND()` 保证精度：

```sql
-- 接单统计（接单率保留 4 位小数）
UPDATE volunteer_profile SET
  total_dispatched = total_dispatched + 1,
  total_accepted   = total_accepted + 1,
  acceptance_rate  = ROUND((total_accepted + 1) / (total_dispatched + 1.0), 4)
WHERE user_id = :userId

-- 平均评分（保留 2 位小数，与前端展示一致）
UPDATE volunteer_profile SET
  avg_rating    = ROUND((COALESCE(avg_rating,0) * COALESCE(total_ratings,0) + :rating)
                       / (COALESCE(total_ratings,0) + 1.0), 2),
  total_ratings = COALESCE(total_ratings, 0) + 1
WHERE user_id = :userId
```

### 3.4 分布式调度锁 + Watchdog 续期

**问题**：多实例部署时，6 个 `@Scheduled` 任务在每台服务器上都会执行，导致同一批订单被重复处理（重复派单、重复超时通知）。

**实现**（`SchedulerLockService`）：

```
tryLock("dispatchQueue", 10s)
  └── SETNX scheduler:lock:dispatchQueue {instanceUUID} 10  → 成功则获锁
       └── 启动 Watchdog 线程，每 3s（10s/3）续期一次：
            └── Lua: if GET key == myUUID then EXPIRE key 10 end

releaseLock("dispatchQueue")
  └── 停止 Watchdog
       └── Lua: if GET key == myUUID then DEL key end  ← 原子校验+删除
```

**TTL 作用**：仅作为崩溃兜底。只要 Watchdog 正常运行，任务执行时间超过初始 TTL 也不会导致锁失效（Watchdog 会持续续期）。

---

### 3.5 WebSocket 跨实例消息转发（Redis Pub/Sub）

**问题**：`UnifiedSessionRegistry` 维护的是**本机 JVM 内**的 WebSocket 连接表。多实例时，盲人连实例 A，志愿者接单的业务逻辑在实例 B 执行，实例 B 查不到盲人的连接，通知丢失。

**实现**：

```
实例B: sendToUser(userId=42, msg)
  ├── 查本机 userSessions → 未找到
  └── PUBLISH ws:messages '{"type":"USER","userId":42,"payload":"..."}'

所有实例（包括B自身）均订阅 ws:messages
  ├── 实例A 收到 → 查本机 → 找到 userId=42 → 发送 ✓
  └── 实例B 收到 → 查本机 → 未找到 → 静默跳过

CS 广播（sendToCs）：直接 PUBLISH，所有实例各自向本机 CS session 投递
```

消息只走 Redis 一次，所有实例并发处理，只有持有该 session 的实例实际发送。

### 3.6 其他并发控制

| 机制 | 位置 | 解决的问题 |
|------|------|-----------|
| **订单乐观锁 + 重试** | `RunOrder.@Version` + `acceptOrderWithRetry()` | 多志愿者同时接同一订单时，只有一个成功，其余重试后感知订单已被接 |
| **HikariCP 连接池** | `application.properties` max=50 | 防止高并发请求耗尽 DB 连接，connection-timeout=30s 保证快速失败 |
| **@Async 有界线程池** | core=20 / max=100 / queue=500 | `@EventListener @Async` 事件处理有上限，防止 OOM |
| **调度线程池独立** | `spring.task.scheduling.pool.size=5` | 6 个调度任务有独立线程，一个慢任务不阻塞其他 |
| **Lettuce 连接池** | max-active=20 | Redis 连接复用，防止每次请求新建连接的开销 |
| **API 限流** | `RateLimitInterceptor` | Redis INCR + 首次 EXPIRE，防止单 IP 滥用 |
| **TTL 驱动派单超时** | `order:dispatch:current:{id}` TTL 30s | 不需要定时器，Redis Key 自然过期即为志愿者超时 |

## 4. 仍然存在的缺陷

### 高风险（生产前必须解决）

| 编号 | 问题 | 影响 | 根因 |
|------|------|------|------|
| R1 | **Redis 单点** | Redis 宕机 → 全部业务中断（派单、位置、认证全部失效） | 当前单机模式，Sentinel 配置已写好但未启用 |
| R2 | **Redis 无持久化（AOF/RDB）** | Redis 重启 → 所有派单队列、位置数据、缓存丢失；DispatchScheduler 会在 5s 内恢复派单队列，但重启期间有空窗期 | 需运维在 `redis.conf` 配置 `appendonly yes`，Spring Boot 应用层无法控制 |
| R3 | **WebSocket 无断线重连下的消息补偿** | 用户断线瞬间发出的 Pub/Sub 消息会永久丢失（Redis Pub/Sub 不持久化） | Pub/Sub 本质是"发完即忘"，无消息队列的持久化和重放能力 |

### 中风险（规模扩大前解决）

| 编号 | 问题 | 影响 | 根因 |
|------|------|------|------|
| R4 | **无缓存预热** | 服务器冷启动时所有 `vol:profile:*` 全部缺失，前几十秒内所有派单请求全部打 DB，可能造成 DB 瞬间高负载 | 没有启动时预热逻辑 |
| R5 | **Profile 缓存无大小控制** | 10 万志愿者 × ~500 字节/条 ≈ 50MB，纯 TTL 失效，无 LRU 淘汰 | `setIfAbsent` + TTL 模式不支持 LRU |
| R6 | **MySQL 无读写分离** | 所有读请求（订单列表、志愿者资料）都打主库 | 单机 MySQL，未配置只读副本 |
| R7 | **乐观锁重试固定间隔 50ms** | 极高并发接单时（大量志愿者同时响应同一订单），线程因重试积压 | `Thread.sleep(50)` 固定间隔，无指数退避 |

### 低风险（可接受或影响极小）

| 编号 | 问题 | 影响 | 说明 |
|------|------|------|------|
| R8 | **通知模板 null 不缓存** | 每次调用不存在的 `eventType` 都会查一次 DB | `disableCachingNullValues()` 是有意设计；正常情况下所有 eventType 都有模板 |
| R9 | **Pub/Sub 广播冗余** | N 个实例中只有 1 个实例最终发送，其余 N-1 个实例都收到消息但静默跳过，小幅增加 Redis 流量 | 实例数 < 10 时影响可忽略；实例数多时可改为"先查 Redis 用户在哪个实例"方案 |
| R10 | **CS 在线人数统计不准** | `getOnlineCsCount()` 只返回本机的 CS 连接数 | 多实例下全局 CS 在线数需遍历所有实例，当前无此接口 |

## 5. 升级路径

### 近期（上线前）

1. **启用 Redis Sentinel**（解决 R1）
   ```properties
   spring.data.redis.sentinel.master=${REDIS_SENTINEL_MASTER:mymaster}
   spring.data.redis.sentinel.nodes=${REDIS_SENTINEL_NODES:host1:26379,...}
   ```
   取消 `application.properties` 中的注释，部署 1 主 + 2 从 + 3 哨兵。

2. **Redis 服务端启用 AOF**（解决 R2）
   ```
   # redis.conf
   appendonly yes
   appendfsync everysec
   ```

3. **缓存预热**（解决 R4）
   启动时将当前活跃志愿者的 profile 批量写入 Redis，消除冷启动 DB 压力。

### 中期（扩容时）

4. **乐观锁指数退避**（解决 R7）
   将 `Thread.sleep(50)` 改为 `sleep(50 * (i+1))`，减少高并发下的无效竞争。

5. **MySQL 主从 + 读写分离**（解决 R6）
   使用 `AbstractRoutingDataSource` 或 ShardingSphere 将只读请求路由到从库。

6. **Profile 缓存加 LRU 上限**（解决 R5）
   Redis 配置 `maxmemory-policy allkeys-lru` 或在应用层限制缓存条目数。

### 远期（百万用户级别）

7. **消息队列替代 Pub/Sub**（解决 R3）
   RocketMQ / Kafka：消息持久化，支持断线重连后消息重放，避免通知丢失。

8. **Redis Cluster**
   数据分片，突破单节点内存 / 吞吐上限。

9. **分布式调度框架**（XXL-Job）
   彻底替代 `@Scheduled` + Redis 分布式锁，提供任务历史、失败重试、分片并行执行。
