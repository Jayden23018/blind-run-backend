# 陪跑轨迹数据留存/删除策略

> 覆盖表：`run_order_track_point`（`RunOrderTrackPoint` 实体），陪跑进行中（`IN_PROGRESS`）盲人+志愿者双方实时上报的经纬度轨迹点，用于走散检测、历史轨迹回放（`GET /api/orders/{id}/track`）、里程/配速统计。

## 1. 数据定性

轨迹点 = 经纬度 + 时间戳 + 关联的用户身份（`userId` + `role`），持续追加写入。

这类数据对应《中华人民共和国个人信息保护法》第 28 条列举的**行踪轨迹**，属于敏感个人信息（其处理对个人可能造成人身、财产损害或人格尊严受侵害，需遵循"最小必要"原则处理）。本项目中该数据额外与"盲人"这一特定身份关联，叠加了第 28 条同条列举的另一类敏感信息（特定身份）。

**处理原则**：
- 仅用于走散检测、纠纷复核、服务质量统计等已声明用途，不做画像、不对外提供。
- 不设置为无限期留存——这是本文档要解决的问题。

## 2. 留存期限

| 项 | 值 |
|---|---|
| 留存期限 | 订单完成（`COMPLETED`/`CANCELLED` 等终态）后 **90 天** |
| 配置项 | `app.track.retention-days`（默认 90，可调） |
| 清理方式 | 每日定时任务硬删除（非软删除，无需保留审计痕迹） |
| 清理粒度 | 按 `recorded_at` 早于 `now - retention-days` 逐行删除，不区分订单状态 |

90 天覆盖常见的纠纷复核窗口（用户投诉/客服介入通常在服务结束后短期内发生），到期后无保留价值，遵循最小必要原则应主动删除而非无限堆积。

**为什么不按订单状态（而是统一按 `recorded_at`）清理**：订单一旦进入 `IN_PROGRESS` 才会产生轨迹点，产生轨迹点的订单必然会走到终态（完成或取消），按时间窗口清理等价于按"订单完成后 N 天"清理，且实现更简单（无需 JOIN `run_orders` 表判断状态）。

## 3. 技术落地

- `RunOrderTrackPointRepository.deleteByRecordedAtBefore(LocalDateTime cutoff)` — Spring Data 派生删除查询。
- `TrackDataRetentionScheduler`（`@Scheduled(cron = "0 0 3 * * *")`，每日凌晨 3 点）— 通过 `SchedulerLockService` 分布式锁防止多实例重复执行，逻辑参照 `OrderTimeoutScheduler`。
- 配置项 `app.track.retention-days=90`，`application.properties` 中声明，默认值内置于 `@Value` 注解（`${app.track.retention-days:90}`），无需强制在测试配置中重复声明。

## 4. 账号注销级联清理（已实现）

`RunOrderTrackPoint` 用裸 `userId` 字段（非 `@ManyToOne` 外键），与 `User` 无数据库级联删除关系，因此清理步骤显式挂在应用层：`UserService.deleteAccount()` → `cascadeDeletePii()` 中调用 `RunOrderTrackPointRepository.deleteByUserId(userId)`，注销时同步清空该用户名下所有轨迹点（含其作为盲人和志愿者两种角色产生的记录），不等待 90 天留存期到期。

若用户主张 PIPL 赋予的删除权但暂不注销账号，仍可人工按 `userId` 手动执行 `DELETE FROM run_order_track_point WHERE user_id = ?`。跟踪见 `docs/ISSUES.md` PR1。
