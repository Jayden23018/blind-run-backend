# 日常部署指南（更新代码 → 上线 → 回滚）

> 本文档面向**日常开发更新**场景：改完代码后，如何把新版本部署到阿里云生产服务器。
>
> 首次服务器搭建（装系统、MySQL、Nginx、SSL）见 [deployment.md](./deployment.md)，本文不重复。

---

## 一、前置条件（只需配置一次）

### 1. SSH 免密登录

部署脚本依赖本地 Mac 到服务器的 **SSH 公钥免密登录**（脚本不交互输入密码，避免密码进入 shell 历史/日志）。

**检查是否已配置：**
```bash
ssh root@47.114.113.171 "echo OK"
```
- 直接打印 `OK`，不问任何密码 → 已配置，跳过
- 还问密码 → 按下面配置

**配置免密（一次性）：**
```bash
# 1. 本地生成密钥（passphrase 必须留空，用 -N "" 强制）
ssh-keygen -t ed25519 -C "mac-deploy" -f ~/.ssh/id_ed25519 -N ""

# 2. 把公钥传到服务器（会让你输一次服务器密码）
cat ~/.ssh/id_ed25519.pub | ssh root@47.114.113.171 "cat > ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"

# 3. 验证（应直接打印 OK，不问密码）
ssh root@47.114.113.171 "echo OK"
```

**服务器侧 SSH 配置**（如果配了免密仍走密码，检查服务器）：
```bash
ssh root@47.114.113.171
# 确保 /etc/ssh/sshd_config 有（去掉前面的 #）：
#   PubkeyAuthentication yes
#   PermitRootLogin yes
grep -E '^PubkeyAuthentication|^PermitRootLogin' /etc/ssh/sshd_config
systemctl restart sshd   # 改完重启
```

### 2. 部署脚本

本地项目根目录放一个 `deploy.sh`（**不提交到 Git**，已加入 `.gitignore`，因含服务器地址等本地配置）。

脚本需具备：构建 jar → 上传 → 数据库迁移检查 → 备份 → 重启 → 健康检查 → 回滚能力。

> 脚本内容较长，由开发者本地维护。换电脑时需重新创建。

## 二、标准部署流程（最常用）

改完代码，**三步**走：

```bash
cd /Users/mac/Downloads/demo

# 1. 提交代码到 GitHub（版本管理，必须先于部署）
git add -A
git commit -m "fix/feat: 简述改动"
git push origin main

# 2. 部署到服务器（一条命令搞定）
./deploy.sh
```

### deploy.sh 做了什么

| 步骤 | 动作 |
|------|------|
| 构建 | `gradlew bootJar -x test` 生成新 jar |
| 上传 | scp 到 `/opt/blindrun/demo-0.0.1-SNAPSHOT.jar.new` |
| 迁移检查 | 自动查 `volunteer_profile.wants_dispatch` 等关键列是否存在 |
| 备份 | 旧 jar 备份为 `.bak.<时间戳>` |
| 重启 | 停服务 → 替换 jar → `systemctl start blindrun` |
| 健康检查 | 轮询 `/v3/api-docs` 最多 60s，失败提示回滚 |

看到 `✅ 部署成功` 即完成。

### deploy.sh 全部参数

| 命令 | 作用 |
|------|------|
| `./deploy.sh` | 完整流程（构建 + 部署 + 健康检查） |
| `./deploy.sh --skip-build` | 跳过构建，用现有 jar |
| `./deploy.sh --only-build` | 只构建不上传 |
| `./deploy.sh --rollback 20260622_204900` | 回滚到指定备份 |

## 三、⚠️ 数据库迁移（改了表结构时必须做）

**判断要不要这步**：本次改动是否修改了 `src/main/java/com/example/demo/entity/` 下的 `@Entity` 类（新增字段、改字段类型、新增表）？

- **没改** → 跳过，直接 `./deploy.sh`
- **改了** → 生产 `ddl-auto=validate`，缺列会导致**启动失败**，必须先手动加列

**手动迁移步骤：**
```bash
ssh root@47.114.113.171
mysql -u root -p
# 在 mysql 里执行：
USE blind_running;
ALTER TABLE <表名> ADD COLUMN <列名> <类型> ...;
-- 例如：ALTER TABLE volunteer_profile ADD COLUMN wants_dispatch BOOLEAN NOT NULL DEFAULT TRUE;
EXIT;
```

然后再 `./deploy.sh`。

> `deploy.sh` 的迁移检查目前只覆盖 `wants_dispatch` 一个列。新增其他列时，要么手动先加，要么在脚本里补检查。

## 四、回滚（部署失败 / 新版本有问题）

### 方法 1：用脚本一键回滚

```bash
# 先看有哪些备份
ssh root@47.114.113.171 "ls -1 /opt/blindrun/demo-0.0.1-SNAPSHOT.jar.bak.*"

# 回滚到指定备份（时间戳从上面列表选）
./deploy.sh --rollback 20260622_204900
```

### 方法 2：手动回滚

```bash
ssh root@47.114.113.171
cd /opt/blindrun
systemctl stop blindrun
cp demo-0.0.1-SNAPSHOT.jar.bak.20260622_204900 demo-0.0.1-SNAPSHOT.jar
systemctl start blindrun
exit
```

> **回滚安全说明**：回滚旧 jar 后，已新增的数据库列会保留（旧代码不认识这些列，但 `validate` 模式下**多余列不报错**，只有**缺失列**才报错）。所以"加了列 → 部署 → 回滚"是安全的。

## 五、常用运维命令

```bash
# 看服务实时日志（Ctrl+C 退出）
ssh root@47.114.113.171 "sudo journalctl -u blindrun -f"

# 看最近 100 行日志
ssh root@47.114.113.171 "sudo journalctl -u blindrun -n 100 --no-pager"

# 看服务状态
ssh root@47.114.113.171 "systemctl status blindrun"

# 手动重启（不部署新代码，比如改了 env.conf 后）
ssh root@47.114.113.171 "systemctl restart blindrun"
```

### 修改生产配置（env.conf / JVM 参数）

生产配置在 `/opt/blindrun/env.conf` 和 systemd unit 文件的 `ExecStart`（`-D` JVM 参数）。改完必须重启：
```bash
ssh root@47.114.113.171
sudo systemctl daemon-reload    # 改了 unit 文件才需要
sudo systemctl restart blindrun
```

详见 [CLAUDE.md → Production Deployment](../CLAUDE.md)。

## 六、排查：部署后服务起不来

先看日志定位：
```bash
ssh root@47.114.113.171 "sudo journalctl -u blindrun -n 100 --no-pager"
```

常见报错：

| 报错关键字 | 原因 | 解决 |
|-----------|------|------|
| `Schema-validation: missing column [xxx]` | 缺数据库列 | 手动 `ALTER TABLE` 加列（见第三节） |
| `Failed to bind properties under 'xxx'` | `application.properties` 配置错或缺 env var | 检查 env.conf / `-D` 参数 |
| `Port 8081 was already in use` | 旧进程没停干净 | `systemctl stop blindrun; pkill -f demo-0.0.1; systemctl start blindrun` |
| `Communications link failure` | MySQL/Redis 没起 | `systemctl status mysql redis-server` |

日志看不懂就回滚保平安：
```bash
./deploy.sh --rollback <最近的时间戳>
```

## 附：一次性把 deploy.sh 提交问题说明

`deploy.sh` 被 `.gitignore` 忽略，不上传 GitHub。原因：
- 含服务器 IP、路径等本地/环境特定配置
- 换电脑时需重新创建（可从本文档脚本逻辑复刻）

如需团队共享部署逻辑，建议：
- 把脚本里的"配置区"变量改为读环境变量（`SERVER`、`REMOTE_JAR` 等从 `.env` 读）
- 这样脚本本身可提交，敏感配置留在各自的 `.env`（仍 gitignore）
